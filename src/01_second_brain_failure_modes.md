# 1. Second Brain은 왜 유지되지 않는가

Second Brain이 실패하는 이유는 사용자가 게으르기 때문이 아니다. 기록과 정리를 같은 순간에 요구하기 때문이다. 개발자는 하루 동안 회의, 디버깅, 코드 리뷰, 학습, 장애 대응, 개인 아이디어를 오간다. 이때마다 폴더를 고르고, 태그를 붙이고, 링크를 만들고, MOC 위치를 고민하면 기록 자체가 줄어든다.

기존 흐름은 대개 다음 순서다.

```text
작성
 ↓
태그 선택
 ↓
폴더 선택
 ↓
링크 생성
 ↓
정리
```

이 방식은 작성 시점에 미래의 활용 맥락을 예측하라고 요구한다. 하지만 어떤 기록이 프로젝트 문서가 될지, 학습 노트가 될지, 질문 큐가 될지는 시간이 지나야 분명해진다. 그래서 이 책은 작성 시점의 책임을 줄이고, 정리 시점의 Agent 루프를 강화한다.

## Agent가 해결해야 하는 문제

Agent가 해결해야 할 문제는 "예쁜 폴더 구조 만들기"가 아니다. 핵심은 구조화 부채를 낮추는 것이다.

- 흩어진 Daily Note 조각을 독립 노트, 프로젝트 문서, 질문 큐로 분리한다.
- 기존 노트와의 중복을 찾아 병합 또는 링크 후보를 만든다.
- 새 노트가 고립되지 않도록 관련 링크를 제안한다.
- MOC와 graph 품질 지표를 갱신한다.
- 모든 변경을 diff와 report로 남겨 사용자가 검토할 수 있게 한다.

완전 자동 정리는 매력적으로 보이지만 위험하다. vault는 개인 지식의 장기 저장소이므로 Agent는 먼저 제안하고, 신뢰도 높은 작업만 자동 적용해야 한다.
