# 1. Second Brain은 왜 유지되지 않는가

Second Brain이 실패하는 이유는 사용자가 게으르기 때문이 아니다. 기록과 정리를 같은 순간에 요구하기 때문이다. 개발자는 하루 동안 회의, 디버깅, 코드 리뷰, 학습, 장애 대응, 개인 아이디어를 오간다. 이때마다 폴더를 고르고, 태그를 붙이고, 링크를 만들고, MOC 위치를 고민하면 기록 자체가 줄어든다.

기존 흐름은 대개 다음 순서다.

```text
작성
 ↓
태그 선택
 ↓
폴더 선택
 ↓
링크 생성
 ↓
정리
```

이 방식은 작성 시점에 미래의 활용 맥락을 예측하라고 요구한다. 하지만 어떤 기록이 프로젝트 문서가 될지, 학습 노트가 될지, 질문 큐가 될지는 시간이 지나야 분명해진다. 그래서 이 책은 작성 시점의 책임을 줄이고, 정리 시점의 Agent 루프를 강화한다.

## 작성 시점의 분류는 왜 자주 틀리는가

개발자의 하루 기록은 생성 당시에는 경계가 흐리다. 장애 대응 중에 발견한 Redis miss는 프로젝트 이슈이지만, 동시에 캐시 전략 학습 노트의 재료이기도 하다. 회의에서 나온 "검색 API p95를 낮추자"는 말은 action item이지만, 나중에는 성능 개선 사례 연구의 일부가 될 수 있다. React Server Component의 `cache()` 동작을 다시 보겠다는 메모는 질문으로 시작하지만, 며칠 뒤에는 frontend architecture MOC의 핵심 링크가 될 수 있다.

작성자가 이 모든 미래 용도를 즉시 판단해야 하면 두 가지 일이 생긴다.

첫째, 기록이 늦어진다. 생각을 붙잡아야 할 순간에 저장 위치를 고르기 시작한다. 둘째, 분류가 보수적으로 변한다. 어디에 둘지 애매한 기록은 daily note나 inbox에 방치되고, 결국 다시 찾아보기 어렵다. 시간이 지나면 방치된 기록은 "정리해야 하는 빚"이 되고, 빚이 커질수록 Obsidian을 여는 일 자체가 부담이 된다.

Agent 기반 Second Brain은 이 문제를 다른 방식으로 본다. 작성 시점의 분류 정확도를 높이려 하지 않는다. 대신 작성 시점에는 아주 작은 protocol만 지키고, 정리 시점에 더 많은 문맥을 가진 Agent가 후보를 만든다. 오늘의 기록, 지난주 프로젝트 문서, 이미 존재하는 concept note, graph의 연결 상태를 함께 본 뒤에 위치와 링크를 결정하는 편이 더 정확하다.

## Second Brain의 운영 실패 모드

Obsidian vault가 커질수록 실패는 UI 문제가 아니라 운영 문제가 된다.

| 실패 모드 | 증상 | 원인 | Agent가 할 수 있는 일 |
| --- | --- | --- | --- |
| Inbox 적체 | `inbox/`와 daily note만 계속 늘어난다 | 정리 비용이 기록 비용보다 크다 | archive queue와 주기적 처리 |
| 중복 노트 | 같은 개념이 여러 이름으로 존재한다 | 파일명 검색만으로 기존 노트를 찾기 어렵다 | alias, semantic search, 병합 후보 |
| 링크 부채 | 좋은 노트가 graph에서 고립된다 | 작성 시점에 관련 노트를 기억하지 못한다 | related link와 backlink 제안 |
| MOC 부패 | 목차가 오래되어 실제 vault와 다르다 | 새 노트 추가와 MOC 갱신이 분리되어 있다 | MOC diff 생성 |
| 프로젝트 맥락 손실 | daily note 안에 결정과 근거가 묻힌다 | 작업 기록과 프로젝트 문서가 연결되지 않는다 | project note 병합과 source link |
| 질문 방치 | 궁금증은 많지만 research queue가 없다 | 질문을 action으로 바꾸는 루프가 없다 | question note와 weekly review |

이 표에서 중요한 점은 Agent가 모든 문제를 "정답으로 고친다"가 아니라는 것이다. Agent의 1차 역할은 부채를 보이게 만드는 것이다. 어떤 노트가 고립되었는지, 어떤 주제가 중복되었는지, 어떤 질문이 반복되는지, 어떤 MOC가 실제 graph와 어긋났는지를 report로 드러내야 한다.

## Agent가 해결해야 하는 문제

Agent가 해결해야 할 문제는 "예쁜 폴더 구조 만들기"가 아니다. 핵심은 구조화 부채를 낮추는 것이다.

- 흩어진 Daily Note 조각을 독립 노트, 프로젝트 문서, 질문 큐로 분리한다.
- 기존 노트와의 중복을 찾아 병합 또는 링크 후보를 만든다.
- 새 노트가 고립되지 않도록 관련 링크를 제안한다.
- MOC와 graph 품질 지표를 갱신한다.
- 모든 변경을 diff와 report로 남겨 사용자가 검토할 수 있게 한다.

완전 자동 정리는 매력적으로 보이지만 위험하다. vault는 개인 지식의 장기 저장소이므로 Agent는 먼저 제안하고, 신뢰도 높은 작업만 자동 적용해야 한다.

## Agent 도입의 세 가지 원칙

첫 번째 원칙은 capture와 curation을 분리하는 것이다. 사람은 빠르게 capture하고, Agent는 나중에 curation한다. 이 분리를 지키지 않으면 Agent를 도입해도 사용자는 여전히 작성 시점에 구조를 고민하게 된다.

두 번째 원칙은 모든 자동화를 diff로 표현하는 것이다. "React Query 노트를 업데이트했다"는 보고만으로는 부족하다. 어느 파일의 어느 섹션에 어떤 문장을 추가했고, 어떤 링크를 새로 만들었는지 확인할 수 있어야 한다. 개발자에게 신뢰 가능한 자동화는 결국 diff를 남기는 자동화다.

세 번째 원칙은 graph를 결과물이 아니라 feedback loop로 보는 것이다. 링크가 많다고 좋은 vault가 아니다. 모든 노트가 hub note에만 연결되어도 graph는 복잡해 보이지만 실제 탐색성은 나빠진다. Agent는 graph를 보고 다음 archive 정책을 조정해야 한다. orphan note가 많으면 related link 추천을 강화하고, hub note가 과도하면 더 구체적인 중간 노트를 만들고, broken link가 늘면 rename/move 정책을 점검한다.
